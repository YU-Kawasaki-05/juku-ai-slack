# ユーザー像 — juku-ai-slack-bot

## 1. ユーザー種別一覧

| ID | 種別 | 説明 | 想定人数 | 利用頻度 |
|----|------|------|---------|---------|
| U-01 | 生徒 | 塾の受講生。SlackでBotに質問する主対象 | 数人〜数十人 | 週数回〜毎日 |
| U-02 | スタッフ | 塾の指導員・運営担当者。生徒チャンネルで質問・管理画面利用 | 数人 | 毎日 |
| U-03 | 管理者 | システム全体の管理者。管理画面の全機能を使用 | 1〜2人 | 週数回 |
| U-AI | AI Coding Agent | 仕様→実装の翻訳者（Claude Code等） | - | 常時 |

---

## 2. ペルソナ詳細

### U-01: 生徒

- **役割**: 学習中に生じた疑問をBotに質問する
- **主な利用目的**: 数学・英語等の問題の解き方を聞く、理解を確認する
- **利用シーン**: 自宅学習中（夜・休日含む）にスマホ/PCのSlackから質問
- **利用デバイス**: スマートフォン（主）、PC（副）
- **ITリテラシー**: 低〜中（Slackの基本操作は可能。Botのメンション方法を案内する必要あり）
- **主要タスク**:
  1. Botをメンションして質問する
  2. 画像（問題文スクショ・手書き）を添付して質問する
  3. 返答に対して追加質問する（スレッド内）
- **ペインポイント**: 夜や休日に質問できない。スタッフの返信待ちで学習が止まる
- **成功体験**: いつでも個別対応された回答がすぐ得られる。途中式や考え方が丁寧に説明される

---

### U-02: スタッフ

- **役割**: 生徒の指導サポート・レポート管理・Bot利用状況の確認
- **主な利用目的**: 生徒チャンネルでの補足回答、管理画面でのレポート管理・エラー確認
- **利用シーン**: 塾のPC/スマホからSlackおよび管理画面にアクセス
- **利用デバイス**: PC（主）、スマートフォン（副）
- **ITリテラシー**: 中（Slack・ブラウザ操作は問題なし。DB操作等は不要）
- **主要タスク**:
  1. 生徒チャンネルでBotの回答を確認・補足する
  2. 管理画面で月次レポートを作成・編集する
  3. エラーや異常な利用状況を確認する
  4. 「スタッフ向けに」「指導方針として」等の指示でBotにスタッフ向け補足を引き出す
- **ペインポイント**: Bot回答の品質確認が面倒。エラー発生時に原因が分からない
- **成功体験**: レポートを更新するだけでBotの回答品質が向上する。エラーが一覧で確認できる

---

### U-03: 管理者

- **役割**: システム全体の設定・管理・コスト監視
- **主な利用目的**: チャンネル紐付け設定、利用状況・コストの把握
- **利用シーン**: PCの管理画面からアクセス
- **利用デバイス**: PC
- **ITリテラシー**: 高（システム管理に習熟。ただしコード変更は不要）
- **主要タスク**:
  1. 新規生徒登録・チャンネル紐付け設定
  2. トークン消費・推定コストの確認
  3. エラーの対応済みマーク・メモ記入
  4. レポートのembedding再生成
- **ペインポイント**: コストが見えない。エラーが発生してもユーザーから報告されるまで気付けない
- **成功体験**: ダッシュボードで全状況を一覧把握できる

---

### U-AI: AI Coding Agent

- **役割**: 仕様→実装の翻訳者（Claude Code等のコーディングエージェント）
- **主な利用目的**: コード生成・コードレビュー・リファクタ・drift検出
- **必要な情報**:
  - 機能IDと実装ファイルの対応（`@implements FR-XX, AC-XX-XX`）
  - 入出力の型・バリデーション規則
  - エラーハンドリング規約（エラーコード一覧・ユーザー向け文言）
  - セキュリティ境界（Slack署名検証・Supabase RLSの適用範囲）
  - 非同期処理の境界（ACKまでとジョブ処理の責任分離）
- **苦手なこと**:
  - 暗黙の前提・口頭で決まったルール
  - 複数ファイルにまたがる暗黙の依存
  - [仮決定]のまま実装が進んだ箇所の意図
- **扱い方針**:
  - 実装ファイル冒頭に `// @implements FR-XX, AC-XX-XX` を必須化
  - 仕様変更は `/要件定義/12_change_request` 経由で行う
  - `[仮決定]` タグの項目は実装前に確定させる
