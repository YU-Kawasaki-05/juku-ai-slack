# Phase 1 サマリー — juku-ai-slack-bot

## 1. エグゼクティブサマリー

Slack連携型・生徒別AI学習支援Botシステム。生徒ごとの専用Slackチャンネルで、月次レポート・画像・スレッド履歴を参照しながらAIが個別対応する。インフラはVercel + Supabase（pgvector）。主UIはSlackで独自チャットUIはMVPスコープ外。管理画面で生徒・チャンネル・レポート・エラー・利用状況を管理する。

## 4. 主要決定事項

| DEC-ID | 決定内容 |
|--------|---------|
| DEC-04 | 主UIはSlack。独自チャットUIはMVPスコープ外 |
| DEC-06 | SlackチャンネルIDを信頼の基点。channel_nameは表示用のみ |
| DEC-07 | AI回答トーンは「生徒本人向け・やさしく丁寧」を基本 |
| DEC-08 | RAGは3層構造（プロフィール要約 + チャンク検索 + スレッド履歴） |
| DEC-09 | Slack Events APIのACKとAI処理を分離（非同期ジョブ） |
| DEC-10 | 画像はVision modelに直接渡す（OCR先行はMVP後回し） |
| DEC-11 | チャンネル直下: メンション必須。スレッド内: 登録済みならメンション不要 |

## 7. Phase 2（外部設計）への引き継ぎ

**設計時に検討が必要な事項:**
- DB詳細設計（インデックス・RLS・外部キー制約）
- API設計（`POST /api/slack/events` の詳細フロー・`/api/admin/*` 全エンドポイント）
- Supabase pgvectorのインデックス設定（IVFFlat vs HNSW）
- 非同期ジョブ基盤の詳細（Supabase job table実装 vs Inngest/QStash）
- LLMモデルの確定とコスト試算
- embeddingモデルの確定
- Supabase StorageのRLS設定（画像への不正アクセス防止）
- 管理画面のUIフレームワーク選定

**未解決の課題:**
- 1つのperson_idへの複数チャンネル紐付け時のRAGフィルタ方法
- スレッド長期要約（FR-20）のトリガー条件と実装方法
- レポート作成をAI補助にするか手入力のみにするか
