# FR-26: 適応型ワークド例題（Backward Fading）

## 概要

P(mastery) < 0.3 の「苦手」状態にある生徒に対して、認知負荷理論（CLT）と
Worked Example Effect に基づく4段階 Backward Fading を適用する。
GPTutor 研究（2023）が示す「チャット単純回答 → 答え探し行動」リスクを防ぎ、
自己説明を促進することで長期記憶への定着を狙う。（DEC-23）

## 前提条件

- FR-05 の direct モード（P < 0.3）でのみ適用
- 試験前モード中は Fading フェーズを適用しない（直接指導のみ）

---

## 4段階 Backward Fading フェーズ

| フェーズ | P(mastery) 目安 | 例題タイプ | 説明 |
|---------|---------------|-----------|------|
| F1 完全例題 | P < 0.3 (初回) | 全ステップ公開 | 条件・行動・目的を全て示す |
| F2 補完問題 | P ≈ 0.3〜0.4 | 最後のステップを空白 | 「ここまで来たら最後の式は？」 |
| F3 ヒント付き演習 | P ≈ 0.4〜0.6 | ヒントのみ提示 | ステップを示さず、方向性だけ |
| F4 独立演習 | P ≥ 0.6 相当 | FR-05 Socratic に移行 | Backward Fading 終了 |

フェーズは連続スレッド内の attempt_count と signal 履歴から動的に決定する。

---

## 例題フォーマット（「条件・行動・目的」フレームワーク）

各ステップを「条件→行動→目的」の3要素で記述する。
認知負荷を分散するため、ステップ間で必ず生徒の入力を待つ（一度に全て送らない）。

```
【例題: 二次方程式の解法】

--- ステップ1 ---
条件: ax²+bx+c = 0 の形式が与えられている
行動: a, b, c の値を特定する
目的: 次のステップで代入できる値を確定するため

理解できたら「次」と返信してね
```

### Slack 向け Split-Attention 対策

数式と説明を別々に出力すると視線移動コストが発生する。
コードブロックで同じ行に記述する：

```
```
x = (-b ± √(b²-4ac)) / 2a   // b²-4ac が 判別式 D にあたる
```
```

## 最低例題数ルール

- 独立演習に移行する前に、同トピックで最低 **2例題** を提示する
- 1例題のみでは「転移学習（Transfer Learning）」が成立しない（研究上の知見）
- 例題の数値は「認知負荷が少ない数」を優先する（例: 11, 12 など計算しやすい数値）

---

## 自己説明プロンプト（Self-Explanation）

例題提示後、以下4タイプから文脈に合う1つを挿入する。

| タイプ | 目的 | 例 |
|-------|------|-----|
| 原理の言語化 | なぜその操作をするかを言わせる | 「ここで両辺を m で割ったのはどうしてだと思う？」 |
| 条件チェック | 使える条件を確認させる | 「この公式が使えるのは問題文にどんな条件があるから？」 |
| 反事実 (IF-NOT) | 他の手順との違いを考えさせる | 「もし最初に y について解いていたらどうなってたと思う？」 |
| エラー分析 | 典型的な誤答の論理ミスを見つけさせる | 「ある生徒がここで〇〇という式を立てて間違えました。どこが違う？」 |

---

## Expertise Reversal Effect 対策

P(mastery) が上昇するにつれ、ワークド例題が **逆効果**（Expertise Reversal Effect）になる。

- P ≥ 0.5: 例題提示を段階的に削減し、ヒントのみに移行
- P ≥ 0.6: FR-05 Socratic モードへ切り替え（Backward Fading 終了）
- P ≥ 0.8: FR-05 confirmation モードへ移行
- フェーズが上がった後は古いフェーズに戻さない（Backward Fading は順方向のみ）

---

## ビジネスルール

- BR-26-01: 同一トピックで最低2例題を提示してから独立演習へ移行する
- BR-26-02: 1メッセージに複数ステップを詰め込まない（Transient Information Effect 防止）
- BR-26-03: ステップ間で必ず生徒からの入力（または「次」反応）を待つ
- BR-26-04: 数式と説明を分離しない（コードブロックで同行表記）
- BR-26-05: P(mastery) ≥ 0.5 になった時点で例題提示を減らし始める
- BR-26-06: 自己説明プロンプトは質問で終わり、その後にヒントを付け足さない（BR-05-XX と同じ原則）

## 受入基準（AC）

### AC-26-01: P<0.3 で完全例題が提供される
```gherkin
Given 生徒Aの「二次方程式」の p_mastery が 0.18 である
When 生徒Aが二次方程式の解法を質問する
Then F1（完全例題）フォーマットで「条件・行動・目的」が全ステップ示される
And 1ステップずつ区切られ、生徒の返信を待つ形式になっている
```

### AC-26-02: 同トピックで2例題後に補完問題へ移行
```gherkin
Given 同一スレッドで「二次方程式」の完全例題を2回提示済みである
When 生徒Aが3回目の関連質問をする
Then F2（補完問題）が提示される（最後のステップが空白になっている）
```

### AC-26-03: 自己説明プロンプトが挿入される
```gherkin
Given F1 例題の提示後
When 次の返信が生成される
Then 4タイプの自己説明プロンプトのいずれかが含まれる
And 自己説明プロンプトの後にヒントが続かない
```

### AC-26-04: P≥0.5 で例題なしのSocraticに切り替わる
```gherkin
Given 生徒AのP値が上昇して 0.55 になった
When 次の返信モードを選択する
Then Worked Example（F1/F2）は使用されない
And FR-05 Socratic モードが適用される
```

### AC-26-05: 数式と説明が同行表記になっている
```gherkin
Given 数式を含む例題が生成される
When Slackに送信される
Then 数式と説明がコードブロック内の同一行または隣接行に記載されている
And 数式のみ単独送信されていない
```
