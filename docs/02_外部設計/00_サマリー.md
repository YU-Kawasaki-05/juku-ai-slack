# Phase 2 サマリー — juku-ai-slack-bot

## 1. 設計概要

DB12テーブル（pgvector含む）、APIエンドポイント18本（Webhook1本 + Admin API17本）、管理画面13画面を設計。認証はSupabase Auth（メール+パスワード）。Slack BotはService Role経由でDBに直接アクセス。テストは Unit/Integration/E2Eの3層、P0の全ACをテスト対応表に網羅。

## 3. 主要設計判断

| # | 判断 | 決定内容 | 理由 | 代替案 |
|---|------|---------|------|--------|
| 1 | フロントエンド | Next.js App Router | Vercelとの親和性・SSR/API Route統合 | Remix, SvelteKit |
| 2 | UIライブラリ | shadcn/ui + Tailwind | カスタマイズ性・型安全 [仮決定] | MUI, Chakra UI |
| 3 | 状態管理 | TanStack Query [仮決定] | サーバーステート管理・キャッシュ | SWR |
| 4 | embedding次元数 | 1536 [仮決定] | OpenAI text-embedding-ada-002想定 | 1024（Anthropic等） |
| 5 | pgvectorインデックス | IVFFlat (lists=100) [仮決定] | 数万件規模で十分・設定が簡単 | HNSW（精度優先） |
| 6 | BotのDB認証 | Service Role（RLSバイパス） | シンプル・サーバーサイドのみ | 専用Botロール |
| 7 | テストツール | Vitest + Playwright | TypeScript親和性・高速 | Jest + Cypress |

## 5. Phase 3（技術設計）への引き継ぎ

- **フレームワーク確定**: Next.js App Router 前提で進めてよいか確認
- **非同期ジョブ基盤**: Supabase job table実装の具体的なpolling/triggerパターン設計
- **LLMモデル確定**: モデル名・コスト試算・プロンプトテンプレート設計
- **embeddingモデル確定**: 次元数が変わる場合はDB設計に影響
- **Slack Appの設定**: Bot User、Event Subscriptions、OAuth Scopes の具体的設定
- **ディレクトリ構成**: Next.js App Routerのファイル配置設計
- **環境変数一覧**: 全環境変数の整理
