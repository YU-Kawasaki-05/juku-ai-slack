# 非機能要件 — juku-ai-slack-bot

## 1. パフォーマンス（NFR-PERF）

| ID | 要件 | 基準値 | 測定方法 | 優先度 |
|----|------|--------|---------|--------|
| NFR-PERF-01 | Slack Events APIへのACK応答時間 | **3秒以内**（Slackの必須要件） | Vercel Function Logs | 最高 |
| NFR-PERF-02 | AI回答生成〜Slack返信完了（非同期ジョブ） | 15秒以内（通常の質問） | ai_usage_logs.latency_ms | 高 |
| NFR-PERF-03 | 画像付き質問のAI回答完了 | 30秒以内 | ai_usage_logs.latency_ms | 高 |
| NFR-PERF-04 | 管理画面ページ初期表示 | LCP 3.0秒以内 | Lighthouse / Vercel Analytics | 中 |
| NFR-PERF-05 | 管理画面API応答時間（一覧取得） | 500ms以内（p95） | Vercel Logs | 中 |
| NFR-PERF-06 | RAGベクトル検索応答時間 | 200ms以内 | クエリログ | 中 |

> NFR-PERF-01は設計で解決済み（DEC-09: ACKとAI処理の分離）。

| ID | 要件 | 対応 | 優先度 |
|----|------|------|--------|
| NFR-SEC-01 | Slack署名検証の必須実装 | FR-01 BR-01-01〜02 | 最高 |
| NFR-SEC-02 | 生徒間のデータ越境防止 | channel_id→person_id解決の徹底 | 最高 |

| NFR-SEC-05 | AIからのシステム情報漏洩防止 | システムプロンプトに情報制限を明記 | 高 |
| NFR-SEC-06 | raw_errorのマスキング | 管理画面表示時にAPIキー等をマスク | 高（MVP後） |

| NFR-SEC-09 | HTTPSの強制 | Vercelで自動対応 | 最高 |
| NFR-SEC-10 | ログへの個人情報保存制限 | APIキー・機密情報の保存禁止 | 高 |

## 3. 可用性（NFR-AVL）

| ID | 要件 | 基準値 | 備考 |
|----|------|--------|------|
| NFR-AVL-01 | システム全体稼働率 | 99.5%/月（月間ダウンタイム3.6時間以内） | Vercel / Supabaseの可用性に依存 |
| NFR-AVL-02 | AI APIが利用不可の際の対応 | ユーザーに適切なエラーメッセージを返す（AI_RATE_LIMITED等） | FR-11参照 |
| NFR-AVL-03 | ジョブ失敗時のリトライ | max_attempts=3でリトライ | FR-04参照 |
| NFR-AVL-04 | Slack ACKの保証 | ジョブ登録失敗時もACKを返す（Slackからのリトライを防ぐ） | FR-01参照 |

## 4. スケーラビリティ（NFR-SCL）

| ID | 要件 | 基準値 | 備考 |
|----|------|--------|------|
| NFR-SCL-01 | 生徒数増加への対応 | 生徒数200人まで追加設定なしで動作 | - |
| NFR-SCL-02 | Vercel Functionの同時実行 | Vercelのデフォルト同時実行数内で動作 | MVP時は小規模のため問題なし |
| NFR-SCL-03 | pgvectorのスケール | チャンク数20,000件までインデックスなし再設定不要 | IVFFlat (lists=100) で対応 |
| NFR-SCL-04 | 非同期ジョブ基盤の移行性 | Supabase job tableからInngest/QStashへの移行を最小コードで可能にする | FR-04参照 |
