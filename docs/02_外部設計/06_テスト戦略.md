# テスト戦略 — juku-ai-slack-bot

## 1. テストピラミッド目標

```
          E2E (5%)
         /Playwright/
       ─────────────
      Integration (25%)
     /Vitest + local Supabase/
   ─────────────────────────────
        Unit (70%)
      /Vitest/
```

- Unit : Integration : E2E = 70 : 25 : 5
- 行カバレッジ目標: **80%**
- ブランチカバレッジ目標: **70%**
- カバレッジ除外: `src/types/`, `supabase/migrations/`, `*.config.ts`, `*.d.ts`

---

## 2. レイヤー別テスト方針

### 2.1 Unit テスト

**対象**: ビジネスロジック・ユーティリティ・バリデーション・エラー分類
**ツール**: Vitest
**配置**: 対象ファイルと同階層に `*.test.ts`

**必須対象（BR-XXを持つロジック全件）**:
| 対象 | テストファイル例 |
|------|----------------|
| Slack署名検証ロジック | `src/lib/slack/verifySignature.test.ts` |
| 反応制御判定ロジック | `src/lib/slack/shouldReact.test.ts` |
| エラーコード → ユーザー文言マッピング | `src/lib/errors/errorMessages.test.ts` |
| RAGプロンプト構築ロジック | `src/lib/ai/buildPrompt.test.ts` |
| チャンク分割ロジック | `src/lib/rag/chunkReport.test.ts` |
| usage log のコスト計算 | `src/lib/ai/calculateCost.test.ts` |

---

### 2.2 Integration テスト

**対象**: API Routes・DB アクセス・外部サービスのモック
**ツール**: Vitest + Supabase local（`supabase start`）
**配置**: `src/__tests__/integration/`

**必須対象（全P0機能のAPI Route）**:
| 対象 | テストファイル例 |
|------|----------------|
| POST /api/slack/events（署名検証・job登録） | `src/__tests__/integration/slack-events.test.ts` |
| チャンネル紐付け検索 | `src/__tests__/integration/channel-binding-lookup.test.ts` |
| レポートRAGチャンク検索 | `src/__tests__/integration/rag-search.test.ts` |
| 管理画面API（CRUD） | `src/__tests__/integration/admin-persons.test.ts` |
| ジョブ登録・処理フロー | `src/__tests__/integration/job-processing.test.ts` |

**外部API（LLM / Slack）はMSWでモック**:
```ts
// src/__tests__/mocks/handlers.ts
import { http, HttpResponse } from 'msw'

export const handlers = [
  http.post('https://api.anthropic.com/*', () => {
    return HttpResponse.json({ content: [{ text: 'テスト回答' }] })
  }),
  // Slack Web API mock
]
```

---

### 2.3 E2E テスト

**対象**: 主要管理画面フロー（P0機能の主要AC）
**ツール**: Playwright
**配置**: `tests/e2e/`

**必須対象**:
| フロー | テストファイル |
|--------|-------------|
| ログイン / ログアウト | `tests/e2e/auth/login.spec.ts` |
| 生徒作成 → プロフィール入力 | `tests/e2e/persons/create-person.spec.ts` |
| チャンネル紐付け作成 | `tests/e2e/bindings/create-binding.spec.ts` |
| レポート作成 → 公開 | `tests/e2e/reports/create-report.spec.ts` |
| Embedding再生成 | `tests/e2e/reports/rebuild-embeddings.spec.ts` |
| エラー一覧 → 対応済みマーク | `tests/e2e/errors/resolve-error.spec.ts` |

**E2Eでは実際のAI APIは呼び出さない**（MSWまたはモックサーバーを使用）。

---

### 2.4 Visual Regression

MVP時点では**不要**。管理画面のデザインが安定したタイミングで検討。

## 4. テストデータ戦略

- Factory: `src/__tests__/factories/` 配下
  - `PersonFactory.ts`, `ReportFactory.ts`, `ChannelBindingFactory.ts` 等
- Seed: `supabase/seed.sql`（`supabase db reset` 時に自動実行）
- クリーンアップ: Integration テストは各テスト前後に対象テーブルをtruncate（または transaction rollback）

## 5. CI パイプライン

| トリガー | ジョブ | 並列度 | 失敗時 |
|---------|-------|--------|-------|
| PR | Lint + 型チェック + Unit | 1 | Block merge |
| PR | Integration（local Supabase） | 1 | Block merge |
| PR | E2E smoke（主要3フロー） | 1 | Block merge |
| main merge | E2E fullsuite | 2 | Slack通知 [仮決定] |

---

## 6. テスト品質ルール

- **AAAパターン**（Arrange-Act-Assert）を徹底
- テスト名は「〜した場合、〜になる」形式（日本語可）
- AI生成テストには `// @verifies AC-XX-XX` を冒頭に必須
- フレーキーテストはマージ禁止（再現確認してfix or revert）
- 外部API（LLM / Slack）はUnit/IntegrationでMSWモック必須
- E2Eでは本番AI API呼び出し禁止
