# 学習塾向けテキストベースAIチューターにおける例題学習効果と認知負荷理論の応用：2024〜2026年の研究動向と実装ガイドライン

# 序論：認知負荷理論に基づくテキストベースAIチューターのパラダイム転換と学習塾への適用

2024年から2026年にかけて、生成AIを用いた教育支援システム（Intelligent Tutoring Systems: ITS）の設計において、対話型アプローチと認知科学の統合が急速に進展している。特に日本の学習塾という、限られた時間内で高度な学力向上と試験対策が求められる特異な教育環境において、学習者の認知プロセスに最適化されたAIチューターの導入が急務となっている。本報告書は、Slackなどのチャットプラットフォーム上で稼働するAIチューターを構築するにあたり、学習者の習熟度確率（P(mastery)）が0.3未満の新規概念に直面した際のアプローチに焦点を当てる。

学習者のP(mastery)が0.3未満である場合、対象となる概念は学習者にとって完全に新規であり、スキーマ（知識の構造的枠組み）が長期記憶に存在しない状態を意味する。この状態で、従来から推奨されてきたソクラテス式問答法（Socratic questioning）を適用して「自ら答えを導き出す」ことを要求することは、認知負荷理論（Cognitive Load Theory: CLT）の観点から極めて不適切であるというエビデンスが蓄積されている。認知負荷理論は、学習課題そのものが持つ「課題内在性負荷（Intrinsic Cognitive Load）」、不適切な教材設計に起因する「課題外在性負荷（Extraneous Cognitive Load）」、そしてスキーマ構築に寄与する「学習関連負荷（Germane Cognitive Load）」の3つから構成される。P(mastery)が0.3未満の初学者の場合、新規概念そのものがもたらす課題内在性負荷がすでにワーキングメモリの容量を圧迫している。ここでソクラテス式問答法を用いると、存在しない知識を探索するプロセスによって課題外在性負荷が限界を超え、学習は阻害される。

このような初学者に対しては、直接教授法（Direct Instruction）と例題学習（Worked Example）の組み合わせが最も高い学習効果をもたらすことが実証されている。例題学習効果（Worked Example Effect）とは、自力で問題を解かせる代わりに、完全に解法が示された例題を学習させることで、ワーキングメモリの資源を未知の解決策の探索ではなく、問題状態と解決手順の関連付け（スキーマ構築）に集中させる現象である。テキストベースの対話型AIシステムにおいてこの効果を最大化するためには、単に正解を提示するだけでなく、段階的な情報提示、補完問題（Completion Problems）を通じたフェージング、自己説明（Self-Explanation）の誘発、視覚的分割注意（Split-Attention）の回避など、認知負荷を適切に管理する緻密なアーキテクチャが要求される。本報告書では、これら6つの核心的課題に対する学術的裏付けと、2024〜2026年における最新の本番環境稼働システムの分析を網羅的に論述し、実装に向けた具体的なガイドラインを提示する。

# チャット・テキスト環境における例題（Worked Example）の最適構造

チャット環境における例題とは、課題の提示、解決に向けた手順、および最終的な解答を段階的に例示する構造化されたテキスト群を指す。従来の教科書や静的なウェブページ上の例題と異なり、チャットインターフェースは情報の流動性が極めて高く、長文が一度に送信されると画面から押し出されてしまう「一時的情報効果（Transient Information Effect）」が発生しやすい。したがって、SlackベースのAIチューターにおける例題は、認知的チャンク（情報の塊）ごとに分割し、意図的な間（Deliberate pauses）を設けたインタラクティブなステップ・バイ・ステップ構造を取ることが最適である。

最適な例題の構造は、「条件（Condition）」「行動（Action）」「目的（Goal）」の3要素を明確に言語化するフレームワークに依存する。初学者はしばしば「何をしたか（行動）」という表面的な手順のみを記憶しようとするが、長期的な記憶の定着と転移（Transfer）を促すためには、「なぜその操作が有効なのか（条件）」と「そのステップによって何を達成しようとしているのか（目的）」の論理を明らかにする必要がある。チャットボットが例題を提示する際、専門家が無意識にスキップしてしまう「意思決定ポイント（Decision Point）」を明示的に分解し、どの公式を適用すべきか、どの数値を代入すべきかという選択の根拠を一つひとつテキスト化することが求められる。

具体的なチャットフローとしては、AIが一度に全ての手順を送信するのではなく、一つの意思決定ポイントごとに解説を行い、生徒からの「次へ進む」という承認（または「理解しました」という短いリアクション）を待つ設計が推奨される。この意図的な一時停止（Mandatory reflection pauses）は、生徒に直前のステップをワーキングメモリ内で処理し、短期記憶から長期記憶へと移行させる時間を与えるための極めて重要なメカニズムである。さらに、AIが「思考のGPS」として生徒の認知プロセスを完全に肩代わりしてしまうリスクを防ぐため、システムは「コンパス」として方向性を示すにとどめ、次のステップへ進む前に生徒自身の認知的な処理を挟む余白を残す構造が理想的である。

| 例題構成要素 | チャットテキストにおける実装要件 | 認知負荷への影響と期待される効果 |
| --- | --- | --- |
| **原則の提示（Principle）** | 適用する定理や公式を明記し、なぜそれがこの問題に適切なのかを言語化する | 既存のスキーマとの接続を促し、学習関連負荷（Germane Load）を高める |
| **条件の明示（Condition）** | 問題文のどの部分がそのステップを正当化しているのかを指摘する | 問題の表層的特徴と深層構造の紐付けを強化する |
| **行動と目的（Action & Goal）** | 「何を計算したか」と「それによって何が求まるのか」をセットで提示する | 盲目的な手続きの暗記を防ぎ、論理的推論能力を育成する |
| **意図的な一時停止（Pause）** | ステップごとに送信を区切り、生徒からのアクション（入力やボタン押下）を待つ | 一時的情報効果を防ぎ、ワーキングメモリの枯渇を回避する |

# 

長大なプロセスを細かいステップに分割することは、後述する補完問題（Completion Problems）へのシームレスな移行を可能にする基盤となる。数学や物理などの要素相互作用（Element Interactivity）が高い領域では、一回のチャット送信につき1〜2ステップに限定し、生徒のワーキングメモリがオーバーフローしないように制御することが、システム設計における絶対的な原則となる。

# 補完問題（Completion Problems）と適応的フェージングの実装

完全に解法が示された例題（Full Worked Example）から、生徒が自力で解く独立演習（Independent Problem）へと移行する際、その中間に補完問題（Completion Problems）を挟むことが極めて有効である。補完問題とは、問題の解決手順が部分的に示されており、残りの手順を学習者自身に完成させる形式の課題である。認知負荷理論において、補完問題は解答の初期段階での過剰な認知負荷（どのように解き始めるべきかという迷い）を削減しつつ、学習者に残りのステップを完成させるための深い情報処理を要求するという「心理的均衡」を提供する。

AIチューターにおける補完問題の実装は、サポートを段階的に減らしていく「フェージング（Fading）」というプロセスを通じて行われる。研究によれば、最初の手順を生徒に解かせる「前方フェージング（Forward Fading）」よりも、最後の手順から徐々に生徒に委ねていく「後方フェージング（Backward Fading）」の方が、特に初学者に対しては効率的であり、ワーキングメモリの負荷を抑えながら転移学習の成績を高めることが実証されている。

後方フェージングのプログレッションは、段階的に生徒の認知的責任を増加させるように設計される。

| フェーズ | 学習課題の形式 | AIチューターの出力構造 | 生徒に要求されるタスク |
| --- | --- | --- | --- |
| **第1段階** | 完全な例題（Full Example） | 全ての手順（例：Step 1〜4）と解説を提示する。 | 提示された手順を読み、理解し、自己説明を行う。 |
| **第2段階** | 後方フェージング補完問題（軽度） | Step 1〜3の過程を提示し、最後のStep 4のみを空欄にして問う。 | 既に構築された論理の延長として、最終ステップ（Step 4）を計算・推論する。 |
| **第3段階** | 中間フェージング補完問題（重度） | Step 1のみを提示し、以降のStep 2〜4の道筋を問う。 | 最初の手掛かりを元に、後半のステップを自力で導出する。 |
| **第4段階** | 独立演習（Independent Problem） | 問題文のみを提示し、手順の提示を完全に排除する。 | 全てのステップを自力で解決し、知識の自動化を図る。 |

# 

チャットベースの環境においてこのプログレッションを実装する場合、AIは生徒の正答率や解答にかかる時間をリアルタイムで監視し、動的にフェージングの度合いを調整する「適応的フェージング（Adaptive Fading）」のアルゴリズムを持つ必要がある。例えば、生徒が第2段階の補完問題でエラーを出した場合、AIは直ちに正解を提示するのではなく、段階的なヒントを与えたり、再度第1段階に近い支援レベルへと後退させる。逆に、迅速に正答を導き出した場合はサポートを急速にフェードアウトさせ、「専門性の逆転効果（Expertise Reversal Effect）」——すなわち、既に知識を持つ学習者に対する過剰な解説が逆に認知負荷を高め、学習を阻害する現象——を防がなければならない。

日本の学習塾においては、伝統的に「基本問題（Kihon Mondai）」の解法を黒板で体系的に示し、その直後に構造が同じで数値だけが異なる「類題（Ruidai）」を解かせるという「構造化された問題解決（Structured Problem Solving）」の教授法が定着している。Slackボットの実装においても、この文化的・教育的文脈を継承し、最初の例題を提示した直後に提供する補完問題を「類題」として位置づけ、段階的に穴埋め要素を増やしていく設計が、生徒の学習体験とシームレスに適合する。

# 自己説明効果（Self-Explanation Effect）を誘発するプロンプト設計

例題や補完問題が持つ最大の落とし穴は、学習者がテキストを表面上なぞっただけで理解したつもりになる「説明深度の錯覚（Illusion of Explanatory Depth）」に陥ることである。生成AIが流暢で権威ある解説を出力するほど、生徒は自分自身の理解度を過大評価し、実際に自力で解こうとすると手が止まってしまうという現象が頻発する。この問題に対処するための最も強力な認知的触媒が「自己説明（Self-Explanation）」である。例題をただ読むだけでなく、自らの言葉で「なぜその手順が妥当なのか」を説明した学習者は、パッシブに読み流した学習者に比べて、数学的証明や物理の成績が飛躍的に向上することが一貫して示されている。

テキストベースのAIチャットボットが自己説明を誘発するためには、システムプロンプトに「メタ認知的な協同調整（Co-regulation）」の役割を組み込む必要がある。具体的には、例題のステップが完了するごとに、AI側から能動的に質問（プロンプト）を投げかけ、生徒に解答の論理的根拠を記述させる。研究によれば、自己説明は学習者の前頭前野のメタ認知ネットワークを活性化し、盲目的なパターンマッチングではなく、深い概念的理解を構築する。

自己説明を誘発するための具体的なプロンプト設計には、以下のような多様なアプローチが考えられる。

| プロンプトの類型 | AIチャットボットからの具体的な発話例（Slackへの出力例） | 誘発される認知プロセスと目的 |
| --- | --- | --- |
| **原則・理由の言語化** | 「次のステップに進む前に教えてください。ここで両辺を質量*m*で割ったのは、どのような目的を達成するためですか？」 | 単なる「行動（割った）」ではなく、その背後にある「目的（加速度の分離）」を言語化させる。 |
| **条件・適用可能性の確認** | 「この場面でこの公式を使えるのは、問題文にどのような条件が書かれているからですか？」 | 表面的な数値だけでなく、公式の適用条件（Condition）への注意を促す。 |
| **反事実的推論（IF-NOT）** | 「もし最初のステップで*x*ではなく*y*について解いていた場合、計算過程はどう変わっていたと思いますか？」 | 代替ルートの検討を通じ、現在採用している解法の優位性や必然性を深く理解させる。 |
| **誤答の自己修正と分析** | 「ある生徒がここで〇〇という式を立てて間違えました。この生徒の考え方のどこに論理的な飛躍があるか、説明できますか？」 | エラーの分析を通じて、自身のスキーマに潜む誤解を事前に修正する。 |

対話型AIがこれらのプロンプトを提示し、生徒が不十分な自己説明（例えば「公式に代入しただけ」といった浅い回答）を返した場合、AIは即座に正解を与えるのではなく、さらに深く掘り下げるプロンプトを返すように設計されるべきである。例えば「それは『何をしたか』ですね。では『なぜそうしたのか』、その理由となる数学的な原則を教えてください」といった形でさらなる推論を要求する。これにより、AIは単なる「思考のGPS（答えへ直行させるツール）」から、「思考のコンパス（学習者自身の方向感覚とメタ認知を養うツール）」へと昇華される。

# Slack環境における視覚的分割注意の回避と数式表現の最適化

認知負荷理論における重大な警告の一つに「分割注意効果（Split-Attention Effect）」がある。これは、互いに関連する図や数式、およびそのテキスト解説が空間的・時間的に離れて配置されている場合、学習者が両者を視覚的に行き来し、頭の中で統合しなければならないために生じる無駄な認知負荷（課題外在性負荷）を指す。

Slackのようなチャットプラットフォームでは、教育に特化した専用アプリケーションと異なり、LaTeXのような高度な数式レンダリングがネイティブで完全にサポートされておらず、Markdown形式のプレーンテキストに依存せざるを得ない場面が多い。このような環境において、数式が記述されたメッセージと、その数式を解説するメッセージが分割して送信されたり、生徒が過去のメッセージへスクロールして戻らなければならない設計は、著しい認知負荷の増大を招く。

この課題を克服するためには、解説テキストと数式を物理的に統合させる「空間的接近の原則（Spatial Contiguity Principle）」をSlackのUI上の制約の中で最大限に実現するベストプラクティスを導入する必要がある。第一に、数式の変形プロセスを示す際、「上の式において」といった指示代名詞を使用することを避け、計算プロセスのすぐ隣にインラインコメントの形式で解説を配置する。SlackのMarkdown環境において、コードブロック（バッククォート3つ）を利用し、数式と解説を等幅フォントで美しく一体化させることが、分割注意を回避する最も実用的な解決策である。

Step 2: 運動方程式の立式と加速度の分離

F = m * a          // ニュートンの第2法則に基づく基本公式を立てる
a = F / m          // 目的：加速度(a)を求めるため、両辺を質量(m)で割る
a = 120 / 15       // 条件：問題文より F=120N, m=15kg を代入する
a = 8              // 計算結果：加速度は 8 m/s^2 となる

このように、数式の展開と「なぜその操作が行われたか（自己説明のトリガーとなる論理）」をインラインで並置することにより、学習者は数式とテキストを頭の中で統合する視覚的・認知的作業から解放され、操作の意味そのものの理解にワーキングメモリを集中させることができる。

さらに、計算における認知負荷そのものを減らすテクニックとして、日本の算数・数学教育における「11の力（The power of 11）」などの手法をAIの生成ロジックに組み込むことも有効である。これは、アルゴリズムや解法の手順そのものを教える例題において、計算自体（乗算や除算）の複雑さがワーキングメモリを占有しないよう、意図的に「11」などの極めて計算しやすい数値を例題に用いる手法である。AIが例題を動的に生成する際、P(mastery)が低い段階では計算の負荷（Intrinsic Load）を意図的に下げ、手順の理解に特化させた数値を採用するようプロンプトを制御することが、分割注意の回避と相まって劇的な効果を生む。

# 独立演習への移行基準と最適な例題数

例題から独立演習（自力での問題解決）へ移行するタイミングは、学習効率を左右する決定的な要因である。例題の提示数が少なすぎればスキーマの構築が不十分となり、多すぎれば熟達者の学習を阻害する逆効果（専門性の逆転効果）を生む。学習塾のカリキュラム設計において「いくつの例題を提示すべきか」という問いに対しては、認知科学の研究（Gick & Holyoak, 1983; Reed & Bolstad, 1991）が明確な指針を示している。単一の例題だけでは概念の転移（Transfer）を促進するには不十分であり、最低でも2つの異なる文脈の例題を提示することが、深い概念的理解に不可欠である。さらに、単純な例題の後に複雑な例題を提示するシークエンスが最も一貫して高いパフォーマンスを生み出すことが実証されている。

これをAIチューターの動的なP(mastery)アルゴリズムと統合する場合、固定の例題数ではなく、学習者のパフォーマンスベースの移行基準（Adaptive Fading）を実装することが推奨される。

移行基準のアルゴリズム構築においては、以下のような段階的なしきい値の設定が効果的である。

| 習熟度確率 (P(mastery)) | システムの介入モードと提供する課題数 | 提供する課題の形式と構造 | 目的と認知負荷の管理 |
| --- | --- | --- | --- |
| **0.0 - 0.3** | 直接教授法（高サポート）
必須例題：最低2問 | 基本問題と応用問題の完全な例題（Full Example）。インライン解説付き。 | 初期スキーマの構築。課題外在性負荷の徹底排除と、未知の手順に対する不安の払拭。 |
| **0.3 - 0.5** | 適応的フェージング（中サポート）
必須補完問題：1〜2問 | 例題と同一構造の「類題」を用いた後方フェージング補完問題。 | 認知的均衡の保持。自己説明の定着を図り、解答の最終ステップを自力で導出させる。 |
| **0.5 - 0.7** | 適応的フェージング（低サポート）
必須補完問題：1問 | 前方フェージング、または最初の手掛かりのみを提供する重度の補完問題。 | メタ認知の監視と部分的な自立的解決。複雑性の高い文脈での概念の適用。 |
| **0.7以上** | 独立演習（ノーサポート）
実践問題：連続正答まで | 独立した実践問題。解説は解答提出後にのみ提供。 | 学習内容の自動化（Automaticity）の獲得と、新しい文脈への転移（Transfer）の確認。 |

このアプローチにより、学習者の進捗に合わせた動的かつパーソナライズされた支援が可能となる。P(mastery)が0.3未満の場合、最低でも2つの完全な例題を提示するルールをシステムにハードコードし、その後、自己説明プロンプトへの回答の質や補完問題の正答率に基づいて、P(mastery)の値を更新しながらフェージングを進行させる。システムが学習者のエラーパターンや自己説明の停滞から認知的な過負荷を検知した場合、即座に前段階（より詳細な例題提示やヒントの増量）へと後退する適応性が、LLMベースのチューターの最大の強みとなる。

# 2024〜2026年における最新のLLMチューター本番環境からの知見

2024年から2026年にかけての大規模言語モデル（LLM）の教育現場への導入は、概念実証（PoC）の段階を越え、実際の教室や学習塾での大規模な本番環境デプロイメントへと移行した。これらの稼働データは、対話型AIチューターの設計に関する極めて重要な実証的洞察を提供している。

特筆すべきは、離散数学の学部コースに導入されたLLMチューター「GPTutor」の大規模な実証研究である。このシステムは、自由形式で質問に答える「対話型チャットボット（Socratic chatbot）」と、学生の解答に対して局所的なフィードバックを提供する「構造化されたレビュー機能（Embedded feedback）」の両方を備えていた。148名の学生を対象とした分析の結果、チャットボットによる自由な対話サポート単体では、学習者は単に答えを求める行動（Answer-seeking behavior）に傾倒しやすく、宿題の成績は一時的に向上したものの、その後の独立した試験でのパフォーマンス向上には結びつかなかったという重大な結果が報告されている。一方で、学習者自身の解答作業に根ざした構造化されたフィードバックを利用した場合、成績低下との負の相関は見られず、むしろ知識の確実な定着に寄与したことが確認された。この知見は、Slackボットが単なる「自由記述の質問応答マシーン」として機能することの危険性を強く示唆しており、AI側から構造化された例題や補完問題を主導して提供し、回答フォーマットを制御する設計の重要性を決定的に裏付けている。

また、AIとの対話において学習者の認知負荷をリアルタイムで測定する「NeuroChat」の研究では、脳波（EEG）を用いて学習者の認知負荷とエンゲージメントをリアルタイムで監視し、LLMの出力の複雑さを調整する閉ループ適応メカニズムが検証された。この研究は、LLMが不用意に長文や複雑すぎる解説を出力することが、学生の認知的過負荷（Cognitive strain）を引き起こし、学習効果を損なうリスクを指摘している。テキストベースのSlackボットにおいて脳波を測定することは不可能であるが、この知見は、チャットボットの応答文字数を厳密に制限し、情報を小さくチャンク化して提示することの正当性を支持している。

日本の教育市場においても、特有の進化が見られる。BenesseやRecruit（スタディサプリ）といった教育事業者は、単なる汎用AIの導入ではなく、日本の教育カリキュラムに最適化された生成AIの統合を進めている。これらのプラットフォームでは、完全なフリートーク型AIではなく、日本の伝統的な「基本問題・類題」のパラダイムを維持しつつ、生徒の記述や学習履歴に基づいてAIが動的に例題の難易度を調整したり、自己説明を促すハイブリッド型のシステム開発が進行している。

さらに、Surge9に代表される次世代の適応型学習プラットフォームでは、LLMの推論能力と厳格な状態管理（State Management）を組み合わせることで、学習者の正答率、自信度、タスクの所要時間などを監視し、例題のフェージングを完全に自動化する「適応的マスタリーエンジン（Adaptive Mastery Engine）」が実装されている。これにより、AIボットは一時的なチャットのやり取りを超越し、長期的な学習者の認知モデル（Learner Model）を背後で構築しながら、個別の認知負荷に合わせた最適なタイミングでの「介入」と「サポートの撤回」を実現している。

# 結論

東京の学習塾におけるSlackベースのAIチューター構築に向けた本調査は、P(mastery)が0.3未満の初学者に対する指導戦略として、直接教授法と例題学習の統合が、認知科学的根拠に基づく最適な選択であることを明確に示している。最新の研究動向と実運用データは、単なる対話機能の提供が学習者の依存を生むリスクを指摘し、構造化された認知支援の必要性を強調している。

システム設計における核心的な推奨事項は以下の通りである。

第一に、例題の構造は、一時的情報効果を防ぐために意味的チャンクごとに分割し、「条件・行動・目的」のフレームワークを用いて段階的に提示すること。専門家が省略しがちな意思決定ポイントを明示し、生徒のワーキングメモリの枯渇を防ぐ意図的な一時停止（Pause）を必ず実装する。

第二に、独立演習への移行前に必ず補完問題（Completion Problems）を挟むこと。日本の「類題」の文脈を活かし、後方フェージング（Backward Fading）のロジックを用いて、学習者の自立的な情報処理能力を段階的に引き出すアルゴリズムを構築する。

第三に、「説明深度の錯覚」を防ぐため、システム側から意図的に自己説明（Self-Explanation）を要求するプロンプトを挿入すること。解答の正誤判定にとどまらず、「なぜその公式を用いたのか」「もし条件が異なればどうなるか」といったメタ認知的監視能力を養う対話を主導する。

第四に、SlackのMarkdown環境特有の分割注意効果（Split-Attention Effect）を回避すべく、インラインコメントとコードブロックを活用すること。数式とその操作理由のテキストを物理的に統合（Spatial Contiguity）し、視線移動に伴う課題外在性負荷を最小化する。

第五に、例題の提示数は最低2問を基準とし、P(mastery)の変動に応じた適応的フェージング（Adaptive Fading）を実装すること。フリートーク型チャットボットが引き起こす「答え探し（Answer-seeking）」の弊害を認識し、あくまでAIを「思考のコンパス」として機能させるべく、構造化された学習フローへの導線を維持する。

本報告書で提示した認知負荷理論に基づくアーキテクチャ設計は、単にAIを用いて作業を効率化するだけでなく、人間の深い認知処理と概念の熟達を支援する、真に教育的な自律型エージェントの実現に寄与するものである。プラットフォームの制約と人間の認知構造の限界を共に理解した上で実装されるAIチューターこそが、次世代の学習支援システムにおける明確な競争優位性となる。
