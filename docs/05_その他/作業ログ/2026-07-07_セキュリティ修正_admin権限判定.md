# 2026-07-07 セキュリティ修正: admin 権限判定を user_metadata → app_metadata へ

## 脆弱性の内容

`requireAdmin()`（Sprint 6 で新設）が `user.user_metadata?.role === 'admin'` で判定していた。
Supabase の `user_metadata` はログイン中のユーザー自身が
`supabase.auth.updateUser({ data: { role: 'admin' } })` で書き換え可能なため、
非管理者スタッフがブラウザから自分を admin に昇格でき、admin 限定の
Embedding 手動再生成（EP-14、LLM 課金発生）を実行できてしまう。

## 修正内容

- `src/shared/lib/auth/requireAdmin.ts`: 判定を `user.app_metadata?.role !== 'admin'` に変更。
  `app_metadata` は Service Role / Admin API でしか変更できない。
  例外（'unauthorized' / 'forbidden'）と戻り値 `StaffContext` は維持
- `src/shared/lib/auth/requireAdmin.test.ts`: 新規。admin 許可 / staff・未設定は forbidden /
  未認証は unauthorized / **user_metadata.role='admin' だけでは昇格できない**（回帰テスト）の4ケース
- `docs/02_外部設計/03_権限設計.md`:
  - ロール定義表を app_metadata.role に修正 + 権限昇格の注意書き
  - 「2.1 ロールの付与手順」を新設（Admin API / SQL の両方を記載。再ログインで JWT 反映）
  - RLS 例の `auth.jwt()->>'role'` は予約 role claim であり app_metadata.role とは別物、
    実権限判定はアプリ層が担う旨を補足

## 運用上の必須事項

- 既存/新規の管理画面アカウントに `app_metadata.role` を付与しないと、
  Embedding 手動再生成が「管理者のみ実行できます」エラーになる（他の管理画面操作は影響なし）
- 付与手順: `docs/02_外部設計/03_権限設計.md` の 2.1 を参照。付与後は再ログインが必要
