# 運用設計 — juku-ai-slack-bot

## 1. ログ設計

### 1.1 出力先

| ログ種別 | 出力先 | 保管期間 |
|---------|--------|---------|
| アプリケーションログ | Vercel Logs（標準出力） | 1日（Hobby）/ 30日（Pro） |
| エラーログ | Sentry + ai_error_logsテーブル | Sentry: 90日 / DB: 無期限 |
| AI利用ログ | ai_usage_logsテーブル | 無期限 |
| Slackイベント受信ログ | slack_event_receiptsテーブル | 30日で定期削除 |
| ジョブ実行ログ | jobsテーブル | 7日で定期削除（completed/failed） |

### 1.2 ログレベル方針

| レベル | 用途 | 例 |
|--------|------|-----|
| ERROR | 復旧不可・即対応が必要 | AI API完全失敗・DB接続不可 |
| WARN | 復旧可能・要確認 | リトライで成功・rate limit |
| INFO | 主要イベント | Slackメッセージ受信・AI回答送信完了 |
| DEBUG | 開発時のみ（本番無効） | プロンプト内容・DBクエリ詳細 |

### 1.3 構造化ログフォーマット

```typescript
// src/lib/logger.ts
type LogEntry = {
  timestamp: string    // ISO 8601
  level: 'ERROR' | 'WARN' | 'INFO' | 'DEBUG'
  feature?: string     // 'FR-01' など
  person_id?: string   // ハッシュ化（生の値を出さない）
  channel_id?: string
  thread_ts?: string
  message: string
  error?: {
    code: string
    message: string
    // stack は本番では出さない
  }
}
```

### 1.4 個人情報マスキング方針

- `person_id`（UUID）: ログに出してよい（直接個人を特定しない）
- 生徒の氏名: ログに出さない
- Slackメッセージ本文: ログに出さない（ai_error_logsのinternal_messageには概要のみ）
- APIキー・Token: 絶対にログに出さない

### 2.2 Sentry設定

```typescript
// sentry.server.config.ts
import * as Sentry from '@sentry/nextjs'

Sentry.init({
  dsn: process.env.SENTRY_DSN,
  tracesSampleRate: 0.1,  // 本番は10%サンプリング
  environment: process.env.NODE_ENV,
  beforeSend(event) {
    // APIキーをマスク
    if (event.request?.headers) {
      delete event.request.headers['authorization']
    }
    return event
  },
})
```

### 3.2 自動停止の仕組み

```sql
-- kill_switchテーブル（Supabase）
CREATE TABLE kill_switches (
  service VARCHAR(50) PRIMARY KEY,
  enabled BOOLEAN NOT NULL DEFAULT true,
  disabled_at TIMESTAMPTZ,
  reason TEXT
);

INSERT INTO kill_switches (service, enabled) VALUES
  ('ai', true),
  ('embedding', true);
```

```typescript
// src/lib/ai/checkKillSwitch.ts
// @implements -
export async function isAIEnabled(): Promise<boolean> {
  if (process.env.ENABLE_AI !== 'true') return false
  const { data } = await supabase
    .from('kill_switches')
    .select('enabled')
    .eq('service', 'ai')
    .single()
  return data?.enabled ?? false
}
```

### 3.3 日次コスト集計バッチ

Vercel Cron（毎日9時）でai_usage_logsを集計 → 閾値超過でSlack通知 + kill_switch更新

### 3.4 暴走防止のレート制限

- person_id単位: 10回/時（Slackメッセージ処理）
- LLM呼び出し: max_tokens必ず指定（デフォルト2048 [仮決定]）
- 1リクエストあたりの画像処理上限: 3枚（FR-06 BR-06-02）

## 5. インシデント対応 Runbook

### S-01: AI API（Anthropic/OpenAI）障害
1. Sentry / ai_error_logsで `AI_RATE_LIMITED` / `AI_RESPONSE_FAILED` 急増を検知
2. Anthropic / OpenAI ステータスページ確認 (`https://status.anthropic.com`)
3. kill_switchで`ai`をdisabled → Slackに「ただいまAIが利用できません」を自動返信
4. 復旧確認後にkill_switchをenabled

### S-02: Supabase障害
1. Vercel Logsでタイムアウト急増を検知
2. Supabaseステータスページ確認 (`https://status.supabase.com`)
3. Slack Botが機能停止（DB接続不可のため）。ユーザーへの通知手段なし（許容）
4. 復旧後、pending状態の未処理ジョブを確認・再実行

### S-03: コスト暴走
1. 日次バッチで月額閾値超過を検知 → Slack通知
2. 即座にkill_switchで`ai`をdisabled
3. ai_usage_logsを確認し、異常な利用源（特定のperson_id・channel_id）を特定
4. 原因（バグ / 攻撃 / 想定外の利用増）に応じて対処

### S-04: Slackの未処理ジョブ蓄積
1. jobs.status='pending'が10件超を検知
2. Vercel Cron実行ログを確認（Cronが動いていない？Functionタイムアウト？）
3. worker処理の失敗原因を調査（ai_error_logsで確認）
4. 必要に応じてjobs.max_attemptsをリセットして再実行
