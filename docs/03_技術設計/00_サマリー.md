## 3. 主要な技術的決定事項

| 決定 | 内容 | 理由 |
|------|------|------|
| Next.js App Router モノリス | SPA + API + CronをNext.jsに一本化 | ソロ開発でのコンテキスト集約 |
| Supabase Service Roleで全DB操作 | APIサーバーはService Roleを使用 | RLSはフォールバック。コードでperson_idフィルタ必須 |
| @implements タグ必須 | 全機能ファイルにFR/ACタグ付与 | Phase 4の機械的整合性チェックを可能にする |
| Slack ACKと処理の分離 | Webhook ACK後にjobsテーブルで非同期処理 | 3秒タイムアウト対策（RISK-09解消） |
| channel_id → person_id解決 | リクエストのchannel_nameを信用しない | 生徒データ越境防止（DEC-06） |
| RAG 3層構造 | プロフィール要約 + チャンクベクトル検索 + スレッド履歴 | トークン効率とコンテキスト品質のバランス |
| kill_switch + ENABLE_AI env | DBテーブルと環境変数の二重停止機構 | コスト暴走対策（S-03） |
| LLMモデル2段構成 | テキスト=haiku、画像/複雑=sonnet | コスト最適化（~$6-10/月、50生徒） |

## 4. 仮決定事項（要確認）

| ID | 内容 | 影響範囲 |
|----|------|---------|
| DEC-03 | 管理者認証 = メール+パスワード | FR-13全体。変更時はAuth設定変更のみ |
| DEC-12 | ジョブ基盤 = Supabase jobs table + Vercel Cron | FR-04全体。変更時はjobsテーブルの設計変更が必要 |
| - | LLMプロバイダ = Anthropic + OpenAI (Embedding) | FR-05, FR-10。抽象化レイヤーで切替可能 |
| - | デフォルトモデル = claude-haiku-4-5 / claude-sonnet-4-6 | コスト計算に影響 |
| - | Embedding次元数 = 1536 | report_chunks.embedding column。変更時はDB再設計必要 |
| - | TanStack Query 5.x | 管理画面のデータフェッチ。サーバーコンポーネントに切替も可 |
| - | max_tokens = 2048 | AI回答長の上限。コストと品質のトレードオフ |

## 5. 未解決の設計課題（P1以降で検討）

| 課題 | 影響 | 対応方針 |
|------|------|---------|
| Vercel Hobby CronはSprint 2以降で制限に当たる可能性 | ジョブ処理が1日1回しか動かない | MVP初期はcron-job.org等で対応 |
| pgvector IVFFlat インデックスの最適パラメータ | チャンク数が増えると検索性能が低下する可能性 | データ蓄積後に調整（nlist値等） |
| スレッド履歴のトークン上限 | 長い会話でコンテキストが溢れる | 直近N件に限定する上限管理が必要 |
| プロフィール要約の鮮度 | レポート更新後に要約が古くなる | 手動再生成（EP-06）で対応。自動化はP1 |
