## 2. 設計思想: features単位の機能凝集

**原則: 「この機能を消したい」ときにフォルダごと消せる構造**

```
配置判断:
1. その機能でしか使わない   → features/{name}/
2. 2つ以上の機能で使う     → shared/ に昇格（使う前に昇格しない）
3. shared/ui は shadcn/ui のみ → カスタムコンポーネントは機能単位
```

## 3. 命名規則

| 対象 | ルール | 例 |
|------|--------|-----|
| コンポーネント | PascalCase | `PersonList.tsx` |
| ロジック・ライブラリ | camelCase | `lookupBinding.ts` |
| Zodスキーマ | camelCase + Schema | `reportSchema` |
| Server Actions | camelCase + Action | `createReportAction` |
| API Route handlers | `route.ts` | `route.ts` |
| テスト | `*.test.ts` | `verifySignature.test.ts` |
| E2Eテスト | `*.spec.ts` | `login.spec.ts` |

## 4. インポートルール

```ts
// tsconfig.json paths
{
  "@features/*": ["./src/features/*"],
  "@shared/*": ["./src/shared/*"]
}
```

- **features間の直接importは禁止**: 必ず `features/{name}/index.ts` 経由
- **バレルエクスポートは `index.ts` のみ**: shared/components/ui を除く
- **相対パスは同一features内のみ許可**: `./lib/verifySignature` ○、`../../features/reports` ✗

## 5. Supabase クライアントの使い分け

```ts
// src/shared/lib/supabase/server.ts — サーバーサイドのみ
import { createClient } from '@supabase/supabase-js'
export const supabaseAdmin = createClient(
  process.env.SUPABASE_URL!,
  process.env.SUPABASE_SERVICE_ROLE_KEY!  // サーバーのみ。.env.localで管理
)

// src/shared/lib/supabase/client.ts — ブラウザ・サーバー両用（認証済みユーザー）
import { createBrowserClient } from '@supabase/ssr'
export const supabaseBrowser = createBrowserClient(
  process.env.NEXT_PUBLIC_SUPABASE_URL!,
  process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!  // NEXT_PUBLIC_= クライアント公開可
)
```

> **SUPABASE_SERVICE_ROLE_KEY は `NEXT_PUBLIC_` を絶対につけない**。
> クライアントバンドルに含まれるとRLSがバイパスされる。
